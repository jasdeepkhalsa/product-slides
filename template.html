Problem: What do users want?

Forget Features! Focus on Activites.

Activity Based Planning by Mike Conte (Microsoft Excel)

For example,

Let's say we want to build a dating website

Most likely, what would a user want to do / achieve?

  • Check out other's profiles
  • Filter profiles by some criteria
  • Post their own profile
  • Find out how much this service costs in comparison to other dating sites

User Activities turn into Features

Continuing the dating website example:

  • Check out other's profiles - homepage contains a search bar & a mixture of recently added profiles and/or popular profiles
  • Filter profiles by some criteria - filter & sorting options after search
  • Post their own profile - "Post Your Profile" menu option in the top navigation
  • Find out how much this service costs - "Pricing" menu option in the top navigation that comes up as a popup, saving the user time!

Usability Testing

No need to theorize about how customers may be using a product - watch them use it!

Observations can be mixed with Activity Based Planning by constantly asking users what were you trying to do?

Problem: How to get a coder to actually make this for you?

“A functional specification describes how a product will work entirely from the user's perspective. It doesn't care how the thing is implemented. It talks about features. It specifies screens, menus, dialogs, and so on.”

— Joel Spoolky

How to Write a Functional Spec?

  • So long as you can write English you can write a functional spec!
  • Write keeping your customer avataars in mind
  • Go page by page, screen by screen and write about everything in DETAIL!
  • Think of all the what ifs e.g:
    • what if the website goes down - what to show?
    • what if there are no matching profiles?
    • what if a customer is gay?
    • what if the user finds no matches, where to take them next?
    • what if somebody wants to post a profile on their daughter/son's behalf?

How to Create Wireframes?

  • Good old fashioned paper and pen is GREAT!
  • Later you can use an online diagramming tool if you want to pretty it up!

Give Your Spec + Wireframes to a Designer

  • Now it's time to make things look pretty
  • Hire a designer/UX designer/front-end coder to create your webpages which will look how you want them to, but won't actually work
  • Check to make sure they're exactly how you want them!

Give those Designs to a Developer

  • Now it's time to make things actually work! WOOHOO! EXCITING!
  • Hire a programmer/coder/developer to turn your specs + designed pages into actually working pages
  • Some programmers also do the design step, saving you from hiring two people
  • Be ready to supply them with anything they need or answer any questions quickly!!!
  • Check up regularly on progress to ensure everything is on track!

After a coder has finished your website or feature — what's next?

Testing, testing, testing!

  • YOU are responsible for QA/testing
  • Go page by page, and test every part of the website is matching the specs you initially wrote
  • Note down any bugs in a spreadsheet and get your coder to go through each one and fix it
  • After something is fixed you may have to re-test it!
  • Remember: if you adjust your spec you may need to follow the entire process again!
  • Now you know how difficult it is to build awesome things!

Tips in Working with Developers

  • Don't tell them HOW to implement, just state WHAT you want
  • Make them your friends and co-workers
  • Provide them with solid logical arguments for all your decisions
  • Break work down into as small chunks as possible - coders like the feeling of progress
  • When reporting bugs provide as much information as possible! Developers are not psychic

Congrats! You're now ready to launch your website or feature!!!

The Workflow Again...

You will need to do this for every new feature(s) you create!

  • Listen to End User Needs & Wants (Activities, Avatars, Usability)
  • Create Functional Specs + Wireframes (Detailed, What Ifs)

  • Develop
  • QA / Testing
  • Launch
